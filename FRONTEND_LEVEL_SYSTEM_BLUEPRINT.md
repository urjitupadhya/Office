# Frontend Level System Blueprint

## Product Direction

The website should feel like a progression game built around a developer journey.
We do not start with GitHub complexity on day one.
We start with a strong visual frontend and a clear sense of growth.

The first playable state is a solo office.
One developer, one room, one desk, one background, one calm scene.
That scene becomes the emotional anchor for the whole product.

## Level Structure

### Level 01: Solo Office

- Base visual: warm personal office background
- Mood: focused, quiet, early-stage builder energy
- Character count: 1
- Goal: prove the visual style and UX loop
- Elevator: visible as a future progression hint, but not yet meaningful

### Level 02: Small Startup Room

- A few desks appear
- More props and activity enter the scene
- GitHub data can begin mapping collaborators into visible teammates

### Level 03: Team Studio

- Distinct work zones
- More contributors represented in the office
- Team energy, movement, and richer HUD states

### Level 04: Multi-Floor Company

- Elevator becomes functional
- Branches, teams, or maturity stages can map to floors
- This is where the full GitHub office fantasy starts to feel complete

## Frontend-First Build Order

1. Build a polished solo office homepage and level presentation.
2. Use mock or placeholder data for repo identity and office status.
3. Lock the visual language, spacing, overlays, and progression storytelling.
4. Add GitHub ingestion after the frontend feels right.
5. Replace placeholder values with live repo, contributor, and activity data.

## What Level 01 Must Show

- The solo office background as the main scene
- A clear headline that explains the product in one sentence
- A visible level system so the user understands growth is the core mechanic
- A roadmap panel that says frontend first, GitHub next
- A simple explanation that the office grows with the project over time

## GitHub Layer Later

Once the frontend is solid, GitHub data should drive:

- repo name and owner identity
- contributor count
- office upgrades
- unlock state for later levels
- activity-based ambience or motion

## Guiding Rule

The interface should make people feel the journey before it asks them to connect data.
The solo office is not just a placeholder.
It is the first chapter of the product.
